# 知音求職助理：中文完整說明文件

## 1. App 簡介

知音求職助理是一個 AI 求職陪跑工具，目標是協助使用者在求職早期降低焦慮、整理自我優勢、釐清職涯方向，並把對話逐步推進到職缺搜尋與下一步行動。

此 App 採用手機介面設計，使用者可以先上傳履歷與求職紀錄，再進入 AI 聊天。AI 會根據使用者輸入、履歷內容與求職紀錄，以分階段方式引導對話。當 AI 判斷需要搜尋職缺時，會觸發內部職缺搜尋流程，並在聊天畫面顯示職缺卡片。

## 2. 適合使用對象

- 正在找工作，但還不確定職涯方向的人。
- 想把履歷內容轉成更清楚求職敘事的人。
- 需要練習面試回答、整理自我介紹的人。
- 想快速查看與自身條件相關職缺的人。
- 想用低壓、陪伴式對話整理求職下一步的人。

## 3. 核心功能

### 3.1 AI 求職對話

使用者進入聊天後，系統會呼叫 `/api/chat`，把對話紀錄、履歷文字與求職紀錄送入 Google Gemini。回覆會以串流方式逐字顯示，讓使用者感覺像即時聊天。

AI 對話不是單純問答，而是求職引導流程。它會先理解使用者狀態，再協助整理亮點、確認偏好，最後推進到職缺建議與下一步行動。

### 3.2 履歷上傳

首頁可上傳履歷檔案。

支援格式：

- `.pdf`
- `.txt`

PDF 會透過 `/api/parse-pdf` 解析成純文字。PDF 大小上限為 5MB。若 PDF 無法解析，使用者需要改用 `.txt` 版本重新上傳。

### 3.3 求職紀錄上傳

首頁可上傳求職紀錄，例如過去投遞紀錄、職缺截圖、面試紀錄或整理好的文字。

支援格式：

- `.txt`
- `.png`
- `.jpg`
- `.jpeg`
- `.webp`

文字檔會直接讀取內容。圖片會轉成 Base64 文字標記後放入對話脈絡，目前屬於 MVP 輔助資訊，實際理解品質取決於後續 AI 模型與提示詞設計。

### 3.4 分階段求職引導

App 會根據使用者送出的訊息數量推進階段。

目前前端階段規則：

- 第 1 階段：前 5 則使用者訊息，偏重自我理解與狀態釐清。
- 第 2 階段：第 6 到第 10 則使用者訊息，偏重條件整理與職缺方向。
- 第 3 階段：第 11 則之後，偏重推薦、策略與下一步行動。

畫面上方會顯示目前階段。當階段切換時，聊天畫面會插入階段分隔提示。

### 3.5 職缺搜尋

AI 可透過特殊格式觸發職缺搜尋：

```text
[SEARCH:關鍵字|地區]
```

前端偵測到此格式後，會呼叫 `/api/search`。後端會嘗試向 104 人力銀行搜尋職缺，最多回傳 5 筆職缺卡片。

職缺卡片包含：

- 公司名稱
- 職缺名稱
- 薪資資訊
- 地點

若 104 阻擋 server-side 請求、回傳非 JSON、沒有資料，或發生網路錯誤，系統會回傳 mock 職缺資料，確保 MVP 流程仍可展示。

### 3.6 使用次數限制

正式環境中，每個瀏覽器每週可免費使用 2 次聊天流程。次數記錄存在瀏覽器 `localStorage`，key 為：

```text
zhiyin_weekly_usage
```

在 `localhost` 開發環境中，系統會略過每週次數限制，方便測試。

### 3.7 每次聊天的搜尋限制

每次聊天最多觸發 2 次職缺搜尋。這是為了避免 AI 在同一輪對話中過度搜尋，也降低 API 與外部服務不穩定造成的影響。

### 3.8 付費提示彈窗

當使用者超過每週免費次數，或點選職缺區域的進階功能按鈕時，畫面會顯示付費提示彈窗。此功能目前是 MVP 介面展示，尚未串接正式付款流程。

## 4. 使用者操作流程

### 4.1 進入首頁

使用者打開網站後，會看到手機樣式的歡迎畫面。首頁包含：

- App 品牌區
- 履歷上傳按鈕
- 求職紀錄上傳按鈕
- 開始聊天按鈕
- 本週剩餘免費次數提示

### 4.2 上傳履歷

1. 點選「上傳履歷」區塊。
2. 選擇 `.pdf` 或 `.txt` 檔案。
3. 上傳成功後，按鈕狀態會更新，並顯示檔名。
4. 若檔案不支援或 PDF 解析失敗，系統會跳出錯誤提示。

### 4.3 上傳求職紀錄

1. 點選「上傳求職紀錄」區塊。
2. 選擇 `.txt`、`.png`、`.jpg`、`.jpeg` 或 `.webp`。
3. 上傳成功後，按鈕狀態會更新，並顯示檔名。

### 4.4 開始聊天

1. 點選開始聊天按鈕。
2. 若仍有免費次數，系統會進入聊天畫面。
3. App 會自動送出初始訊息，請 AI 開始引導。
4. 使用者可在底部輸入框輸入問題或回覆。
5. AI 回覆時，輸入框會暫時停用，避免重複送出。

### 4.5 使用快速回覆

聊天畫面底部有快速回覆按鈕。點選後，文字會填入輸入框，使用者仍需送出訊息。

### 4.6 查看職缺卡片

當 AI 觸發搜尋且 `/api/search` 回傳資料時，聊天畫面會顯示職缺區塊。使用者可查看公司、職稱、薪資與地點。

目前職缺卡片沒有在 UI 中直接開啟外部連結；後端資料結構已有 `url` 欄位，可作為後續功能擴充。

### 4.7 回到首頁

聊天畫面左上方有返回按鈕，可回到首頁。再次開始聊天會重置當前聊天紀錄、階段與本次搜尋次數。

## 5. 管理者與開發者操作

### 5.1 安裝依賴

```bash
npm install
```

### 5.2 設定環境變數

在專案根目錄建立 `.env.local`：

```bash
GEMINI_API_KEY=your_google_gemini_api_key
GEMINI_MODEL=gemini-2.0-flash
```

必要變數：

- `GEMINI_API_KEY`：Google Gemini API key。

可選變數：

- `GEMINI_MODEL`：Gemini 模型名稱。未設定時預設為 `gemini-2.0-flash`。

請勿把 `.env.local` 或任何真實 API key 提交到 GitHub。

### 5.3 本機開發

```bash
npm run dev
```

開啟：

```text
http://localhost:3000
```

### 5.4 建置

```bash
npm run build
```

### 5.5 啟動正式建置

```bash
npm run start
```

### 5.6 程式檢查

```bash
npm run lint
```

### 5.7 部署到 Vercel

1. 將 repo 連接到 Vercel。
2. 在 Vercel Project Settings 設定環境變數：
   - `GEMINI_API_KEY`
   - `GEMINI_MODEL`
3. 部署 main branch。
4. 部署完成後測試首頁、聊天、PDF 上傳與職缺搜尋。

## 6. API 說明

### 6.1 `POST /api/chat`

用途：產生 AI 聊天回覆。

Request body：

```json
{
  "messages": [
    {
      "role": "user",
      "content": "我想找前端工作"
    }
  ],
  "resumeText": "履歷文字，可省略",
  "applicationRecord": "求職紀錄，可省略"
}
```

行為：

- 讀取 `SYSTEM_PROMPT`。
- 將履歷與求職紀錄加入 system context。
- 呼叫 Gemini `streamGenerateContent`。
- 將 Gemini SSE 回覆轉成純文字串流。

常見錯誤：

- 缺少 messages：回傳 400。
- 缺少 `GEMINI_API_KEY`：回傳 500。
- Gemini API 錯誤：回傳 500。

### 6.2 `POST /api/parse-pdf`

用途：解析 PDF 履歷。

Request：

- `multipart/form-data`
- 欄位名稱：`file`

限制：

- 只接受 PDF。
- 檔案不可超過 5MB。
- 若 PDF 沒有可解析文字，會回傳 422。

Response：

```json
{
  "text": "解析後的 PDF 文字"
}
```

### 6.3 `GET /api/search`

用途：搜尋職缺。

Query parameters：

- `keyword`：必填，職缺關鍵字。
- `location`：選填，地區。

範例：

```text
/api/search?keyword=前端工程師&location=台北市
```

Response：

```json
{
  "jobs": [
    {
      "company": "公司名稱",
      "title": "職缺名稱",
      "salary": "薪資",
      "location": "地點",
      "url": "職缺連結"
    }
  ]
}
```

若使用 mock fallback，response 會額外包含：

```json
{
  "mock": true
}
```

## 7. 資料與隱私注意事項

- 履歷與求職紀錄會在聊天時送到後端 API，並加入 Gemini 對話脈絡。
- 本專案目前沒有資料庫，聊天紀錄不會由後端持久保存。
- 使用次數存在使用者瀏覽器的 `localStorage`。
- 若要正式上線，建議加入隱私權政策、資料刪除機制與更完整的使用者告知。

## 8. 已知限制

- 104 職缺搜尋可能被阻擋，因此內建 mock fallback。
- 付費彈窗尚未串接付款。
- 職缺卡片目前未提供外部連結按鈕。
- 圖片求職紀錄目前只以 Base64 形式放入上下文，並非完整 OCR 流程。
- 每週免費次數以瀏覽器 `localStorage` 計算，不是帳號級限制。
- `TASK.md`、`WORKLOG.md`、部分程式中文註解在某些終端可能顯示亂碼，建議後續統一檔案編碼與文字內容。

## 9. 疑難排解

### AI 沒有回覆

請確認：

- `.env.local` 是否存在。
- `GEMINI_API_KEY` 是否正確。
- Gemini API quota 是否足夠。
- `npm run dev` 或 Vercel logs 是否有 500 錯誤。

### PDF 上傳失敗

請確認：

- 檔案是否為 PDF。
- 檔案是否小於 5MB。
- PDF 是否有可選取文字。
- 若是掃描圖片型 PDF，請改用文字檔。

### 職缺都是假資料

這代表 104 回傳非預期內容、阻擋 server-side 請求，或暫時無資料。MVP 會自動 fallback 到 mock data。若要正式產品化，建議改接穩定的搜尋 API 或自建職缺資料源。

### 免費次數用完

正式環境每週 2 次。開發時可在 localhost 測試，或清除瀏覽器 localStorage 中的 `zhiyin_weekly_usage`。
